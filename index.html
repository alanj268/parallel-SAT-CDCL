Summary

We are implementing a high-performance parallel Boolean Satisfiability (SAT) solver based on the Conflict-Driven Clause Learning (CDCL) algorithm. Our goal is to exploit shared-memory multi-core parallelism (and optionally MPI for distributed memory) by simultaneously exploring multiple branches of the decision tree with independent worker threads, while using an efficient inter-thread clause-sharing protocol so that useful learned clauses discovered by one worker immediately prune the search spaces of all others. We will evaluate correctness and scalability against standard SAT-competition benchmarks, comparing parallel speedup relative to a sequential CDCL baseline.

Background

The Boolean Satisfiability problem (SAT) asks: given a propositional formula in Conjunctive Normal Form (CNF), does there exist an assignment of truth values to its variables that satisfies every clause? SAT is NP-complete, yet modern CDCL solvers routinely handle industrial instances with millions of variables due to powerful heuristics and the power of clause learning.

The CDCL algorithm works as follows: 

function CDCL(formula):
    while true:
        propagate()            // unit propagation (BCP)
        if conflict detected:
            if decision level == 0: return UNSAT
            clause = analyze_conflict()   // 1-UIP cut, resolution
            learn(clause)                 // add to clause DB
            backjump(clause)             // non-chronological backtracking
        else if all variables assigned:
            return SAT
        else:
            decide()           // pick unassigned variable (VSIDS heuristic)

Two phases lend themselves naturally to parallelism:

  • Parallel Search (Portfolio): Multiple independent solver instances ("workers") each maintain their own variable assignment, decision stack, and clause database. Each worker uses a slightly different decision heuristic or random seed so that they explore different regions of the search space. The first worker to find SAT (or all workers to find UNSAT) terminates the overall solve.
  • Clause Sharing: When a worker learns a new clause during conflict analysis, that clause can be broadcast to sibling workers. Receiving a short, high-quality clause allows a worker to immediately eliminate branches it has not yet explored, effectively multiplying the pruning power of the entire portfolio.

The clause sharing protocol is the key integration point: workers must be able to publish and consume clauses with minimal synchronization overhead so that the communication cost never becomes the bottleneck.

The Challenge

Parallelizing CDCL is non-trivial for several interrelated reasons:

  • Irregular, data-dependent workload: The depth and shape of the CDCL search tree depends entirely on the input formula and the order in which conflicts are encountered. There is no way to predict in advance how much work each branch requires, making static load balancing ineffective.
  • High communication-to-computation ratio risk: Naïvely broadcasting every learned clause would flood workers with low-quality clauses, causing them to spend more time filtering and integrating new clauses than actually searching. We must implement quality-based filtering (e.g., clause length thresholds, LBD score) and rate-limiting to keep communication overhead low.
  • Shared clause database contention: Shared-memory clause sharing requires lock-free or low-contention data structures (e.g., concurrent ring buffers or epoch-based queues). Fine-grained locking around the clause database can serialize workers and eliminate parallelism gains.
  • Divergent execution: Different workers will encounter conflicts at entirely different points in their search. This leads to divergent memory access patterns that are difficult to predict or prefetch. Cache efficiency per worker is poor compared to sequential solvers.
  • Correctness under concurrent clause learning: A clause added by one worker must be validated against another worker's current variable assignment before being used for propagation. Ensuring consistent state without sacrificing performance requires careful design.

Through this project we hope to learn how the portfolio approach scales with core count, what clause-quality metrics best predict cross-worker pruning value, and whether a shared-memory approach outperforms a message-passing (MPI) approach at various problem sizes.

Goals and Deliverables

Plan to Achieve

  • A fully correct sequential CDCL solver with VSIDS decision heuristics, 1-UIP clause learning, non-chronological backtracking, and restarts.
  • A parallel portfolio solver with N worker threads (targeting N = 2, 4, 8) using shared-memory C++ threading (std::thread).
  • A basic clause-sharing protocol using a lock-free ring buffer; workers filter shared clauses by length (≤ 8 literals) before accepting them.
  • Demonstrated correctness on at least 50 SAT Competition benchmark instances across both SAT and UNSAT categories.
  • A scalability analysis showing speedup vs. thread count on a representative benchmark set, with a goal of achieving ≥ 4× speedup on 8 cores for hard industrial instances (justified by ManySAT results showing super-linear speedups on some instances due to collaborative pruning).

Hope to Achieve

  • LBD (Literal Block Distance) based clause quality filtering to further reduce communication overhead and improve clause sharing utility.
  • Adaptive restart policies that differ per worker (e.g., Luby vs. geometric) to diversify search and reduce redundant work.
  • An MPI-based distributed variant allowing scaling across multiple GHC nodes, comparing performance with the shared-memory baseline.
  • A thorough parameter study of clause-sharing thresholds and their effect on solve time for different benchmark families.

Fallback Goals (if work goes slowly)

  • A correct parallel solver using a simple, coarse-grained locking scheme (even if performance is suboptimal) that still demonstrates speedup on easy benchmarks.
  • A detailed performance analysis of the sequential solver bottlenecks (profiling BCP, conflict analysis, and clause database management) as a foundation for future parallelization work.

Platform Choice

We chose C++17 on shared-memory multi-core x86-64 hardware (GHC cluster) for the following reasons:

  • Performance-critical inner loops: CDCL's unit propagation loop (BCP) is extremely tight and cache-sensitive. C++ gives us the low-level control over memory layout (watched-literal arrays, 2-literal watching scheme) required to keep this loop fast.
  • Rich threading primitives: C++17's std::thread, std::atomic, and std::mutex provide the building blocks for low-overhead shared-memory parallelism without requiring external frameworks.
  • Shared address space for clause sharing: Portfolio parallelism benefits most from a shared address space because workers can read a newly learned clause from a shared buffer without copying it, keeping communication latency very low. A distributed-memory GPU approach would require serializing/deserializing clauses across PCIe, which is less suitable for this irregular workload.
  • Alignment with existing SAT solver ecosystem: The most impactful industrial SAT solvers (MiniSat, Glucose, CaDiCaL, ManySAT) are written in C++. Our implementation choices are therefore directly comparable to the state of the art.
